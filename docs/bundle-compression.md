# 번들 raw 압축 전략

심층 에이전트에 넘길 모달리티 raw를 **무손실에 가깝게 재표현**하기 위한 규칙 문서.
[bundle_parser](../app/services/bundle_parser.py)의 `parse_for_*_agent()`를 아래 압축기로 확장한다.

## 배경

- 관측 데이터는 중복이 심함 — 압축이 곧 정확도 향상임 (반복에 묻힌 희귀 신호가 드러남).
- 심층 분석 입력은 잡당 수만~수십만 토큰까지 커질 수 있어 TPM 병목의 주원인.
- 절단(truncation)은 최후 방어선이고, 기본 전략은 모달리티별 구조화 재표현.

## 실데이터 검증 근거

SDK 데이터셋(`chok-v2-py-sdk/datasets/sn`, MVP 3종 시나리오)으로 규칙 적합성을 확인함.

| 모달리티 | 관측 사실 | 압축 가능성 |
|---|---|---|
| log | UserService 15만 줄, 에러 200건이 타임스탬프만 빼고 완전 동일. info도 `req_id`만 가변, 골격 반복 | 매우 적합 — 수십 개 템플릿으로 축약 |
| metric | `timestamp,value,metric,instance` 15초 간격 CSV. CPU 값이 특정 시각에 `2%→80%` 급변 | 매우 적합 — 순수 수치 통계 |
| trace | CSV에 `parent_span_id`·`duration_us`·`http_status_code`·`service/operation` 완비 | 압축 적합 / 판정은 에이전트 몫 (아래 주의) |

## 모달리티별 압축 규칙

### 1. log — 템플릿화(dedup)

- **방식**: 가변부(시각·`req_id`·숫자 ID)를 마스킹해 동일 패턴을 하나로 묶음.
- **템플릿 키**: `(파일:줄:함수) + 마스킹된 메시지 골격`.
- **산출물(패턴당 1줄)**: `템플릿 · level · ×횟수 · 최초시각 · 최후시각 · 원문 샘플 1줄`.
- **희귀 라인**: 저빈도 패턴은 원문 그대로 보존 — 압축 대상이 아니라 부각 대상.
- **효과**: 15만 줄 → 수십 줄 수준(10~100배). 에러 200건은 `템플릿 ×200`으로 1줄 축약되며 정보 손실 없음.
- **주의**: `username_678`처럼 의미 있는 가변값은 과도 마스킹하지 않음 — 패턴이 갈려도 유지.

### 2. metric — 통계 + 이상점

- **방식**: 시리즈별로 트리거 이전(baseline)·이후(incident) 구간을 나눠 통계 산출.
- **산출물(시리즈당)**: `min · max · mean · p95 · last` + baseline 대비 이탈 구간만 원값.
- **이상점**: baseline `mean ± k·std`를 벗어난 **onset 시각**과 **peak 값·시각**만 명시.
- **서비스 귀속**: 컨테이너별 시리즈(`socialnet_container_cpu` 등)로 범인 서비스 후보를 좁힘.
- **폴백**: raw 파싱 불가 시 원문 통과 (손실보다 안전 우선).
- **효과**: 시계열 나열 제거, LLM 없이 결정적 산출.

### 3. trace — 구간 집계 + exemplar

- **방식**: `parent_span_id`로 호출 트리를 세우고 `(from→to)` 쌍별로 집계.
- **산출물**:
  - `(서비스, 오퍼레이션)`별 호출 수 · 에러 수 · p50/p95/max `duration`.
  - 서비스별 스팬 볼륨 타임라인 (급감·소실 감지용).
  - 가장 느린/에러 트레이스 소수만 원문 전체(exemplar) 보존.
- **효과**: 토폴로지 요약 + 원본 충실도 동시 확보.

> **주의 — 압축과 판정의 분리**
> 트레이스 룰베이스는 **압축(raw→구조화 표)까지만** 담당함.
> kill 계열 장애의 신호는 HTTP 5xx가 아니라 **스팬 볼륨 급감·자식 스팬 소실** 같은 구조적 변화임 (실데이터에서 status_code는 `200`과 공란뿐, 5xx 없음).
> "media-service가 사라졌으니 원인"이라는 **해석은 trace 심층 에이전트(경량 모델)의 몫**으로 남김 — 단순 `5xx 필터` 룰은 범인을 못 짚음.

## 시각 표기 규칙

- **절대 시각 사용** — 상대 오프셋(`T-42s`)은 쓰지 않음.
  - 근거: 리포트가 상대시각으로 근거를 서술하면 사람이 이해하기 어려움. 정확한 시각 전달이 우선.
- **표기 축약**: 날짜·연도는 생략, `HH:MM:SS.mmm`까지 유지 (번들이 단일 윈도 내라 날짜 중복 불필요).
- **핵심 시각 보존**: dedup의 최초/최후, metric의 onset/peak, trace exemplar의 시각은 **정밀 절대값** 그대로 유지.

## 공통 표현 규칙

1. **서비스별 그룹핑** — 원인 서비스 귀속을 도움.
2. **JSON 금지** — 키 반복이 없는 한 줄 텍스트/TSV로 직렬화 (토큰 절감).
3. **프롬프트 캐싱 배치** — `시스템 프롬프트(고정) → 규칙(고정) → raw(가변)` 순서 고정으로 OpenAI 자동 캐싱 수혜.
4. **정답 유출 방지** — title·bundle_id·fileName 제외(D-020/D-021)는 압축 후에도 동일 적용.

## 절단(truncation) — 최후 방어선

- 압축 후에도 입력 상한 초과 시에만 발동.
- **트리거 시각 주변을 우선 보존**하며 절단.
- 절단 발생 사실을 프롬프트에 명시 (에이전트가 부분 관측임을 인지하도록).
